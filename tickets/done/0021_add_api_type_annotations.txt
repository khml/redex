チケット番号: 0021
タイトル: APIドキュメントに引数／戻り値の型注釈を追加する

概要:
ライブラリ内のクラス仕様書（ドキュメント）に記載されているメソッドAPIについて、現在は主に引数名や動作説明のみが記載されている。これを拡張し、各引数および戻り値について期待されるオブジェクトの『型』を明示することで、利用者と実装者双方の理解を深め、誤用を減らす。

背景:
- 現状の `docs/` のクラス・メソッド記載は引数名のみで、期待されるオブジェクトが不明確な箇所がある。
- 実装コード（`lib/redex/` 以下）では複数の独自型（ASTノード、コンテキスト等）を扱っているため、ドキュメントで型を明示する利点が大きい。

目的:
- 各メソッドのパラメータと戻り値に対して型の注釈（期待するクラス/モジュールやインターフェースの説明）を追記する。
- ドキュメントを参照することで、開発者がメソッドの使用方法を型レベルで把握できるようにする。

範囲（対象ファイル）:
- docs/evaluator.md
- docs/parser.md
- docs/tokenizer.md
- 他の `docs/` 内にあるクラス仕様書（該当箇所を列挙して追加）
- 必要に応じて `lib/redex/` 内の主要クラスにYARD形式の注釈を追記する（オプション）

要件／受け入れ基準:
- 各対象メソッドに対して、以下の形式で型注釈が追加されていること。
  - パラメータ名: 期待する型（クラス名／モジュール／インターフェース） - 簡潔な説明
  - 戻り値: 型 - 簡潔な説明
- 型の記述は実装上の具体的なクラス名（例: `Integer`, `String`, `Redex::AST::Node`, `Context` 等）または抽象的なインターフェース（例: "式評価可能オブジェクト"）で表現する。
- 既存の利用例やコードスニペットがある場合は、注釈例として1つ以上掲載する。
- ドキュメント更新後、少なくとも1名のレビュワーが内容を承認すること。


実装タスク:
1. 対象 `docs/` ファイルを洗い出す（該当メソッド一覧化）。
2. 各メソッドについて実装コード（`lib/redex/`）を参照して期待型を特定する。
3. `docs/` に型注釈を追加する（フォーマットは上記の例に準ずる）。
4. 変更内容をレビュー依頼し、指摘を反映。
5. レビュー完了後、TODOを完了に更新してチケットを移動（運用ルールに従う）。

検証／テスト:
- ドキュメントの型注釈は静的チェックされないが、変更によって利用者向けの使い方が明確になっていることを目視で確認する。
- 例示コードがある場合は、簡易スニペットを実際に動かして破綻しないことを確認する（任意）。

影響範囲／リスク:
- ドキュメントの記述のみのため直接的なランタイム影響はないが、型名の誤記が誤解を生む可能性がある。
- 実装に依存する具体名を記載する場合、実装変更時にドキュメント更新が必要となる。
